## 二、数据模型设计

### 2.1 工作流主表 `lesson_workflows`

```go
// models/workflow.go
type LessonWorkflow struct {
    ID          string         `gorm:"primaryKey;type:varchar(64)"`
    UserID      uint           `gorm:"index"`
    Status      string         `gorm:"type:varchar(32);default:'init'"`
    Topic       string         `gorm:"type:varchar(255)"`   // 课程主题
    Subject     string         `gorm:"type:varchar(50)"`    // 学科
    Grade       string         `gorm:"type:varchar(50)"`    // 年级
    Duration    int            `gorm:"default:45"`          // 课时（分钟）
    OutlineJSON string         `gorm:"type:json"`           // 大纲 JSON
    LessonPlan  string         `gorm:"type:longtext"`       // 教案 Markdown
    PPTPath     string         `gorm:"type:varchar(500)"`   // PPT 文件路径
    ErrorMsg    string         `gorm:"type:text"`
    RetryCount  int            `gorm:"default:0"`
    CreatedAt   time.Time
    UpdatedAt   time.Time
    DeletedAt   gorm.DeletedAt `gorm:"index"`
}
```

**设计决策：**
- `OutlineJSON` 用 JSON 类型存储，避免多表 JOIN，状态机每步读写一个字段而非多个表
- `LessonPlan` 用 `longtext` 存 Markdown 原文，前端渲染时不需要反序列化
- ID 用 UUID（`github.com/google/uuid`），便于分布式扩展
- `PPTPath` 只存文件路径，实际文件在 `runtime/ppt/` 目录下，下载时 `c.File()` 直接返回

### 2.2 工作流消息表 `workflow_messages`

```go
type WorkflowMessage struct {
    ID         uint      `gorm:"primaryKey;autoIncrement"`
    WorkflowID string    `gorm:"index;type:varchar(64)"`
    Stage      string    `gorm:"type:varchar(32)"` // outline / plan / ppt
    Role       string    `gorm:"type:varchar(20)"` // user / assistant / system
    Content    string    `gorm:"type:longtext"`
    CreatedAt  time.Time
}
```

**用途：** 记录每个阶段的对话历史。大纲修改时，将历史消息重新构建到 LLM 请求的 messages 数组中，实现多轮对话记忆。

### 2.3 大纲结构体（纯 Go Struct，不落库）

```go
type CourseOutline struct {

    Grade           string          `json:"grade"`
```

**关键设计：SaveOutline 同时更新两个字段：**
```go
func (wm *WorkflowManager) SaveOutline(workflowID string, outline *models.CourseOutline) error {
    jsonBytes, _ := json.Marshal(outline)
    return dao.Db.Model(&models.LessonWorkflow{}).
        Where("id = ?", workflowID).
        Updates(map[string]interface{}{
            "outline_json": string(jsonBytes),
            "status":       models.WFStatusOutlineReady,
            "updated_at":   time.Now(),
        }).Error
}
```

Updates 使用 map 而非 struct，避免 GORM 零值字段被忽略的问题。

## 四、大纲生成服务

### 4.1 整体流程

```
RAG 检索 (Retriever.Search)
     │ 返回 top-5 相关知识块
     ▼
构建 Prompt (buildOutlinePrompt)
     │ 包含：年级/学科/课时要求 + 参考资料 + JSON 格式约束
     ▼
LLM 流式调用 (StreamChat)
     │ 每个 token 通过 SSE 推送给客户端
     ▼
解析 JSON (parseOutlineJSON)
     │ 三重降级：直接解析 → markdown代码块提取 → 首尾花括号截取
     ▼
SaveOutline → status: outline_ready
```

### 4.2 RAG 检索集成

```go
func (s *OutlineService) retrieveContext(topic string) string {
    results, err := s.retriever.Search(topic, "all", 5)
    if err != nil || len(results) == 0 {
        return ""
    }
    return rag.BuildContext(results, 3000) // 限制 3000 字符，避免超 token
}
```

检索结果格式化后拼入 Prompt，让 LLM 在生成大纲时能引用知识库中的史料。

### 4.3 Prompt 设计

```
请为 初中 学生设计一节 历史 课，主题是「商鞅变法」，课时 45 分钟。

以下是来自知识库的参考资料，请结合使用：
【参考资料】
1. [event] 商鞅变法
秦孝公任用商鞅，于公元前356年和前350年两次变法...
...

请严格按照以下 JSON 格式输出课程大纲，不要包含任何其他内容，只输出 JSON：
{
    "title": "课程标题",
    "objectives": {...},
    "key_points": [...],
    "process": [...]
}
```

**约束 LLM 只输出 JSON 的关键手段：**
1. 系统提示明确要求"只输出 JSON"
2. Prompt 末尾给出完整格式示例
3. 解析时做三重降级兜底（见 4.4）

### 4.4 JSON 解析三重降级

LLM 有时会在 JSON 前后加解释文字，或包裹在 markdown 代码块里，因此需要鲁棒的解析：

```go
func (s *OutlineService) parseOutlineJSON(content string) (*models.CourseOutline, error) {
    var outline models.CourseOutline

    // 第一层：直接解析
    if err := json.Unmarshal([]byte(strings.TrimSpace(content)), &outline); err == nil {
        return &outline, nil
    }

    // 第二层：从 ```json ... ``` 代码块中提取
    jsonStr := extractJSONFromMarkdown(content)
    if jsonStr != "" {
        if err := json.Unmarshal([]byte(jsonStr), &outline); err == nil {
            return &outline, nil
        }
    }

    // 第三层：找第一个 { 和最后一个 }，截取中间内容
    start := strings.Index(content, "{")
    end := strings.LastIndex(content, "}")
    if start >= 0 && end > start {
        if err := json.Unmarshal([]byte(content[start:end+1]), &outline); err == nil {
            return &outline, nil
        }
    }

    return nil, fmt.Errorf("无法从 LLM 响应中解析 JSON")
}
```

## 五、大纲对话式修改服务

### 5.1 多轮对话记忆实现

大纲修改支持多轮（"改这个"→"再改那个"），通过以下方式实现记忆：

```go
func (s *OutlineEditService) buildEditMessages(
    currentOutline *models.CourseOutline,
    editRequest string,
    history []models.WorkflowMessage,  // 从 workflow_messages 取出的历史记录
) []llm.Message {

    messages := []llm.Message{
        {Role: "system", Content: outlineEditSystemPrompt},
    }

    // 1. 把当前大纲作为第一条"上下文"注入
    outlineJSON, _ := json.MarshalIndent(currentOutline, "", "  ")
    messages = append(messages, llm.Message{
        Role:    "user",
        Content: fmt.Sprintf("以下是当前的课程大纲：\n```json\n%s\n```", string(outlineJSON)),
    })
    messages = append(messages, llm.Message{
        Role:    "assistant",
        Content: "好的，我已经了解当前大纲的内容。请告诉我您需要修改的地方。",
    })

    // 2. 追加历史对话（本阶段的所有 user/assistant 消息）
    for _, msg := range history {
        if msg.Stage == "outline" && msg.Role != "system" {
            messages = append(messages, llm.Message{
                Role:    msg.Role,
                Content: msg.Content,
            })
        }
    }

    // 3. 追加本次修改请求
    messages = append(messages, llm.Message{
        Role:    "user",
        Content: fmt.Sprintf("请根据以下修改意见修改大纲，输出修改后的完整 JSON：\n%s", editRequest),
    })

    return messages
}
```

**关键设计：** 每次修改请求时，总是把"当前最新大纲"作为对话上下文的起点，而不是把原始大纲作为基准。这样即使 LLM 在某次修改中有偏差，下一次修改总是基于最新状态，不会累积误差。

### 5.2 修改后状态流转

```
修改请求 → outline_editing
    ↓ LLM 返回修改后 JSON
    ↓ SaveOutline (同时写 outline_json + 状态)
outline_ready  ← 无论修改成功或失败都回到 ready
```

如果解析失败，状态回退到 `outline_ready` 而非 `failed`，让用户可以再次发起修改请求。

## 六、教案生成服务

### 6.1 Prompt 结构

教案生成的 Prompt 分三部分：

```
## 课程大纲          ← JSON 格式，包含所有结构化信息
{...}

## 参考资料（来自知识库）  ← RAG 检索结果，提供史料原文
...

## 教案要求           ← 明确格式要求和内容规范
按 Markdown 格式输出，包含：
- 基本信息 / 教学目标 / 重难点
- 五个环节：导入/新授/练习/小结/作业
- 每环节包含：教师活动（含话术）/ 学生活动 / 设计意图
- 板书设计 / 教学反思（留白）
```

### 6.2 系统提示设计

```go
const lessonPlanSystemPrompt = `你是一位拥有20年教龄的春秋战国历史教学名师，擅长编写详细、生动、实用的教案。

你的教案特点：
1. 教师话术自然流畅，能引发学生思考
2. 善于运用史料原文（如《史记》《战国策》《左传》等）辅助教学
3. 注重师生互动，设计启发式问题
4. 教学环节过渡自然，逻辑清晰
5. 兼顾历史知识传授和历史思维培养`
```

"角色扮演"式系统提示能让 LLM 产出质量更高、更符合教学规范的内容。

### 6.3 流式输出的意义

教案通常 2000~5000 字，生成时间较长（约 20~60 秒）。使用 SSE 流式推送，每生成一个 token 就推给前端显示，用户体验与"正在打字"相同，避免长时间白屏等待：

```go
plan, err := ctrl.planService.GenerateLessonPlanStream(
    outline,
    func(text string) error {
        sendWorkflowSSE(c, WorkflowSSEEvent{Type: "text", Content: text})
        return nil  // 返回 error 会终止流
    },
)
```

## 七、PPT 生成服务（核心技术亮点）

### 7.1 技术选型：纯 Go 生成 OOXML，无第三方依赖

**不使用 `unioffice`（需商业授权）、不调用外部 Python 服务**，而是直接用 Go 标准库 `archive/zip` + `encoding/xml` 生成合规的 `.pptx` 文件。

`.pptx` 本质是一个 ZIP 压缩包，内含 XML 文件，遵循 OOXML（ISO 29500）标准：

```
presentation.pptx (ZIP)
├── [Content_Types].xml          ← 文件类型注册表
├── _rels/.rels                  ← 包级关系
├── docProps/app.xml             ← 应用信息
├── docProps/core.xml            ← 文档元数据
├── ppt/
│   ├── presentation.xml         ← 幻灯片列表（引用所有 slide）
│   ├── _rels/presentation.xml.rels
│   ├── theme/theme1.xml         ← 颜色/字体主题
│   ├── slideMasters/            ← 母版（定义全局布局）
│   ├── slideLayouts/            ← 布局（blank 布局）
│   └── slides/
│       ├── slide1.xml           ← 第一张幻灯片内容
│       ├── slide1.xml.rels      ← 引用 slideLayout
│       ├── slide2.xml
│       └── ...
```

### 7.2 两阶段生成流程

**第一阶段：LLM 提取页面结构**

```go
// PPTStructure 定义
type PPTStructure struct {
    Slides []PPTSlideData `json:"slides"`
}

type PPTSlideData struct {
    Type     string   `json:"type"`     // title / content / interactive / summary
    Title    string   `json:"title"`
    Subtitle string   `json:"subtitle,omitempty"`
    Bullets  []string `json:"bullets,omitempty"`
    Notes    string   `json:"notes,omitempty"`
}
```

Prompt 要求 LLM 输出 JSON 格式的页面结构，例如：
```json
{
  "slides": [
    {"type": "title", "title": "商鞅变法", "subtitle": "初中历史 · 45分钟"},
    {"type": "content", "title": "教学目标", "bullets": ["掌握变法核心内容", "理解变法历史意义"]},
    {"type": "interactive", "title": "思考讨论", "bullets": ["商鞅为什么要推行变法？"]},
    {"type": "summary", "title": "课堂小结", "bullets": ["变法意义：富国强兵", "历史影响：统一基础"]}
  ]
}
```

**第二阶段：Go 直接写 OOXML XML**

```go
func (s *PPTService) createPPTX(outputPath string, structure *models.PPTStructure, outline *models.CourseOutline) error {
    f, _ := os.Create(outputPath)
    w := zip.NewWriter(f)
    defer w.Close()

    slideCount := len(structure.Slides)

    // 写入固定结构文件
    writeZipFile(w, "[Content_Types].xml", buildContentTypes(slideCount))
    writeZipFile(w, "_rels/.rels", relsXML)
    writeZipFile(w, "ppt/presentation.xml", buildPresentationXML(slideCount))
    writeZipFile(w, "ppt/_rels/presentation.xml.rels", buildPresentationRels(slideCount))
    writeZipFile(w, "ppt/theme/theme1.xml", themeXML)  // 历史风格：深红 + 金色

    // 动态生成每张幻灯片
    for i, slide := range structure.Slides {
        slideXML := buildSlideXML(slide)  // 根据 type 选择不同模板
        writeZipFile(w, fmt.Sprintf("ppt/slides/slide%d.xml", i+1), slideXML)
        writeZipFile(w, fmt.Sprintf("ppt/slides/_rels/slide%d.xml.rels", i+1), slideRelsXML)
    }
}
```

### 7.3 四种幻灯片类型的 XML 模板

| 类型 | 视觉特征 | 适用场景 |
|------|---------|---------|
| `title` | 深红背景 + 金色大字 | 封面页 |
| `content` | 深红标题栏 + 白底内容 | 知识讲授页 |
| `interactive` | 米黄背景 + 问号符号项目 | 讨论/提问页 |
| `summary` | 深红背景 + 星号项目 | 小结/总结页 |

每种类型在 XML 中通过 `<a:solidFill>` 设置背景色，`<a:rPr>` 设置字体颜色和大小：

```xml
<!-- 标题页示例（深红背景，金色文字） -->
<p:bg>
  <p:bgPr>
    <a:solidFill><a:srgbClr val="8B0000"/></a:solidFill>  <!-- 深红 -->
  </p:bgPr>
</p:bg>
<a:rPr lang="zh-CN" sz="4400" b="1">
  <a:solidFill><a:srgbClr val="FFD700"/></a:solidFill>  <!-- 金色 -->
  <a:latin typeface="Microsoft YaHei"/>
  <a:ea typeface="Microsoft YaHei"/>
</a:rPr>
```

### 7.4 XML 注入防护

幻灯片内容（来自 LLM）写入 XML 时需要 escape，避免破坏 XML 结构：

```go
func xmlEscape(s string) string {
    var b strings.Builder
    xml.EscapeText(&b, []byte(s))
    return b.String()
}

// 使用时：
<a:t>%s</a:t>   →   fmt.Sprintf("<a:t>%s</a:t>", xmlEscape(slide.Title))
```

### 7.5 动态 Content_Types 和 Presentation 引用

幻灯片数量由 LLM 决定（12~18 页），所以文件类型注册表和 presentation 的引用列表需要动态生成：

```go
func buildContentTypes(slideCount int) string {
    var slides strings.Builder
    for i := 1; i <= slideCount; i++ {
        slides.WriteString(fmt.Sprintf(
            `<Override PartName="/ppt/slides/slide%d.xml" ContentType=".../slide+xml"/>`, i))
    }
    return fmt.Sprintf(`<Types>...(固定部分)...%s</Types>`, slides.String())
}

func buildPresentationRels(slideCount int) string {
    var slideRels strings.Builder
    for i := 1; i <= slideCount; i++ {
        slideRels.WriteString(fmt.Sprintf(
            `<Relationship Id="rId%d" Type=".../slide" Target="slides/slide%d.xml"/>`, 10+i, i))
    }
    return fmt.Sprintf(`<Relationships>...(母版引用)...%s</Relationships>`, slideRels.String())
}
```

## 八、SSE 流式接口设计

### 8.1 SSE 响应规范

所有生成接口均采用 Server-Sent Events，原因：
- 单向推送，比 WebSocket 更轻量
- 天然支持 HTTP/1.1 keep-alive
- 浏览器原生 `EventSource` 支持，前端接入简单

```go
// 设置 SSE 响应头
func setSSEHeaders(c *gin.Context) {
    c.Header("Content-Type", "text/event-stream")
    c.Header("Cache-Control", "no-cache")
    c.Header("Connection", "keep-alive")
    c.Header("Access-Control-Allow-Origin", "*")
}

// 推送 SSE 事件（写完立即 Flush）
func sendWorkflowSSE(c *gin.Context, event WorkflowSSEEvent) {
    data, _ := json.Marshal(event)
    fmt.Fprintf(c.Writer, "data: %s\n\n", string(data))
    c.Writer.(http.Flusher).Flush()  // 强制立即发送，不等缓冲区满
}
```

### 8.2 事件类型设计

```go
type WorkflowSSEEvent struct {
    Type       string      `json:"type"`
    Content    string      `json:"content,omitempty"`  // 文本内容
    Data       interface{} `json:"data,omitempty"`     // 结构化数据（大纲/PPT路径）
    WorkflowID string      `json:"workflow_id,omitempty"`
}
```

| type | 触发时机 | Content/Data 内容 |
|------|---------|-------------------|
| `status` | 阶段切换时 | 进度描述文字 |
| `text` | LLM 每生成一个 token | token 文本 |
| `outline` | 大纲生成/修改完成 | `data: CourseOutline` 结构体 |
| `plan` | 教案生成完成 | `content: Markdown` 全文 |
| `ppt` | PPT 文件生成完成 | `data: {path, download_url}` |
| `error` | 任意步骤异常 | 错误描述 |
| `done` | 本次请求结束 | 总结文字 + workflow_id |

### 8.3 错误处理策略

```go
// 错误时：发送 error 事件 + 更新工作流状态
if err != nil {
    ctrl.workflowMgr.SetError(req.WorkflowID, err.Error())  // DB 记录错误
    sendWorkflowSSE(c, WorkflowSSEEvent{Type: "error", Content: "生成失败: " + err.Error()})
    return  // HTTP 连接关闭，但工作流状态已持久化
}
```

工作流状态持久化在 MySQL，客户端断连后可以通过 `GET /status` 查询最后状态，不会丢失已完成的阶段成果。

## 十、关键工程决策总结

| 决策点 | 选择 | 理由 |
|--------|------|------|
| **PPT 生成方案** | 纯 Go OOXML | 无需额外部署，无授权问题，零依赖 |
| **大纲存储格式** | JSON 存入 longtext | 避免额外表设计，读写方便，结构灵活 |
| **流式输出** | SSE（非 WebSocket） | 单向推送场景更轻量，HTTP 原生支持 |
| **LLM JSON 解析** | 三重降级兜底 | LLM 输出不稳定，需要鲁棒解析 |
| **多轮对话记忆** | 每轮重建完整 messages | 简单可靠，避免状态管理复杂度 |
| **错误恢复** | 状态持久化 + 可重入 | 客户端可随时查询状态，支持失败重试 |
| **知识库集成** | 每个步骤都做 RAG** | 大纲/教案都从知识库取史料，内容更准确 |

> **\*\*仅大纲和教案生成做 RAG，PPT 生成基于教案直接提取结构，不再重复检索。**
