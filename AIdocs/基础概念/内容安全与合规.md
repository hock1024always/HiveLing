### 目标：教育场景的“可用”必须建立在“合规可控”之上

你们项目包含学生画像、成绩、教师声音/肖像等敏感信息，且面向未成年人，面试与评审很可能深挖“合规与安全”。工程上建议把内容安全拆成三类：**数据合规**、**生成内容安全**、**数字人授权与防滥用**。

---

### 1）数据合规（学生画像/成绩/对话日志）

#### 1.1 最小权限与角色隔离

- 角色：`student / teacher / admin`
- 权限规则（示例）：
  - 学生只能访问自己的画像与学习记录（或班级汇总的匿名统计）。
  - 老师可访问所教班级的汇总数据与学生画像（按学校规则）。
  - 管理员可审计但需记录操作日志。

#### 1.2 数据脱敏与审计

- 日志与训练数据：
  - 对话日志落库前可做脱敏（姓名、手机号等）。
  - 训练数据导出必须走审批/脱敏流水线（可用离线脚本 + 审计表）。

---

### 2）生成内容安全（对话/教案/PPT）

#### 2.1 策略层：生成前、中、后三级治理

- **生成前**：输入过滤（辱骂、隐私、违法、诱导作弊）。
- **生成中**：系统提示词约束（教材对齐、拒答策略）。
- **生成后**：输出审核（敏感词、事实核验、引用检查）。

#### 2.2 教育场景建议增加的“引用校验”

如果回答宣称来自某教材章节，但 RAG 实际没检索到对应片段，则：

- 降级为“未检索到资料，建议老师核对”，或提示用户补充资料。
- 记录为质量事件，用于后续评测与修正。

---

### 3）数字人授权与防滥用（声音/肖像）

#### 3.1 授权模型（建议最小实现）

- 教师上传声音样本/头像时写入授权表：
  - 授权范围：课程/学校/时间范围
  - 可撤回：一键撤回后禁止再次生成，并对历史内容做下架策略（按你们产品策略）

```sql
CREATE TABLE teacher_media_consent (
  teacher_id VARCHAR(64) PRIMARY KEY,
  voice_sample_url TEXT,
  avatar_image_url TEXT,
  consent_scope JSONB NOT NULL,
  consent_status VARCHAR(16) NOT NULL, -- active/revoked
  updated_at TIMESTAMP NOT NULL DEFAULT NOW()
);
```

#### 3.2 防滥用技术措施（答辩加分）

- 视频加水印/标识“AI 生成”。
- 生成 API 强鉴权 + 频率限制 + 审计日志。
- 对外下载限制与签名 URL（短期有效）。

---

### 4）面试深挖问题准备

- 你们如何处理“学生输入隐私信息”导致模型回显的风险？
- 如果老师撤回数字人授权，你们如何处理已生成的视频？
- 如何防止学生用对话助手“要答案/要作弊”？拒答策略如何设计？
