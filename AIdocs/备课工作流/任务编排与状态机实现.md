# 备课工作流 — 任务编排与状态机实现

## 一、核心思路

备课流水线包含多个耗时且易失败的步骤（RAG 检索、LLM 生成、PPT 渲染），工程上采用**数据库状态机 + 同步 HTTP + SSE 推流**方案，而非异步队列。每个步骤完成后立即持久化，支持断点可查。

---

## 二、状态机拆分（实际代码）

状态常量定义于 `models/workflow.go:11`：

```go
const (
    WFStatusInit              = "init"               // 工作流已创建
    WFStatusOutlineGenerating = "outline_generating" // LLM 正在生成大纲
    WFStatusOutlineReady      = "outline_ready"      // 大纲生成完毕，等待确认或修改
    WFStatusOutlineEditing    = "outline_editing"    // 教师正在对话式修改大纲
    WFStatusPlanGenerating    = "plan_generating"    // LLM 正在生成教案
    WFStatusPlanReady         = "plan_ready"         // 教案生成完毕
    WFStatusPPTGenerating     = "ppt_generating"     // 正在构建 PPTX 文件
    WFStatusDone              = "done"               // 全流程完成
    WFStatusFailed            = "failed"             // 任意步骤失败
)
```

状态流转路径：

```
init
  └─▶ outline_generating ──▶ outline_ready ◀──▶ outline_editing
                                  │
                                  ▼
                         plan_generating ──▶ plan_ready
                                  │
                                  ▼
                         ppt_generating ──▶ done
                                  │
                             （任意步骤）
                                  ▼
                               failed
```

## 四、控制器中的编排逻辑（`controllers/lessonprep.go`）

以生成大纲为例（`LessonPrepController.Start`）：

```go
// 1. 创建工作流记录（init）
wf, _ := ctrl.workflowMgr.CreateWorkflow(0, req.Topic, req.Subject, req.Grade, req.Duration)

// 2. 状态置 outline_generating
ctrl.workflowMgr.UpdateStatus(wf.ID, models.WFStatusOutlineGenerating)

// 3. 记录用户输入到 workflow_messages（stage=outline, role=user）
ctrl.workflowMgr.AddMessage(wf.ID, "outline", "user", req.Topic)

// 4. 流式生成大纲，每 chunk 通过 SSE 推送给前端
outline, err := ctrl.outlineService.GenerateOutlineStream(
    req.Topic, wf.Subject, wf.Grade, wf.Duration,
    func(text string) error {
        sendWorkflowSSE(c, WorkflowSSEEvent{Type: "text", Content: text})
        return nil
    },
)

// 5. 保存大纲 + 状态置 outline_ready
ctrl.workflowMgr.SaveOutline(wf.ID, outline)

// 6. 发送 outline 结构体事件 + done 事件
sendWorkflowSSE(c, WorkflowSSEEvent{Type: "outline", Data: outline})
sendWorkflowSSE(c, WorkflowSSEEvent{Type: "done", WorkflowID: wf.ID})
```

失败时调用 `SetError` 状态置 `failed`，前端轮询 `/api/lessonprep/status` 可感知。

## 五、多轮对话修改大纲（幂等设计）

`outline_editing` 阶段可多次发起，每次：

1. 从 DB 读取最新 `outline_json`（当前版本）
2. 读取 `workflow_messages` 表中 stage=`outline` 的历史对话（最近 N 条）
3. 拼装 messages → LLM 流式输出
4. 解析新大纲 JSON → `SaveOutline` 覆写（同一 workflow_id，非追加）
5. 状态回到 `outline_ready`

覆写而非追加保证同一工作流始终只有一份最新大纲，可随时重新修改而不产生冗余记录。

## 六、SSE 事件协议

所有 SSE 事件格式（`controllers/lessonprep.go:39`）：

```go
type WorkflowSSEEvent struct {
    Type       string      `json:"type"`        // status|text|outline|plan|ppt|error|done
    Content    string      `json:"content,omitempty"`
    Data       interface{} `json:"data,omitempty"`
    WorkflowID string      `json:"workflow_id,omitempty"`
}
```

前端根据 `type` 字段分别处理：
- `text`：追加到内容区（流式输出）
- `outline`：渲染大纲结构卡片
- `plan`：渲染 Markdown 教案
- `ppt`：显示下载按钮
- `error`：显示错误提示
- `done`：解锁下一步操作按钮

## 七、与"重量级工作流引擎"的对比

| 维度 | 本项目（DB 状态机） | Temporal/Argo |
|------|-------------------|---------------|
| 部署复杂度 | 低，仅依赖 MySQL | 高，需额外组件 |
| 重试 | 手动重置状态重新请求 | 框架自动重试 |
| 可视化 | 需自建查询接口 | 内置 Dashboard |
| 适用场景 | 步骤少（≤10）、用户交互型 | 步骤多、纯异步批处理 |

> 本项目属于教师主导的**交互式**工作流（每步等待确认），同步请求 + SSE 推流是合适的工程选择；若后续扩展为批量备课，再迁移至 Temporal。
