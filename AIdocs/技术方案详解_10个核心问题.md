## 1. RAG 数据清洗的核心

### 当前方案

**数据流完整流程**：

```
原始文本(TXT/MD) → DocumentParser.ParseFile() → 切分Chunk → 元数据标注 → MySQL存储 → Milvus向量化
```

**具体实现代码位置**：`backend/services/rag/document_parser.go`

**清洗步骤详解**：

| 步骤 | 代码函数 | 做了什么 | 为什么这样做 |
|------|---------|---------|-------------|
| 1. 格式标准化 | `preprocessHistorical()` | 统一标点符号（全角转半角）、标准化换行 | 避免"。"和"．"被当成不同字符 |
| 2. 结构识别 | 正则匹配 `(卷[一二三四].+\|本纪\|列传)` | 识别章节标题并标记为`【章节】` | 保留文档结构，方便后续按章节检索 |
| 3. 实体预标注 | 正则匹配人名模式 | 识别"齐桓公"、"孔子"等已知人物 | 为后续NER提供候选集 |
| 4. 段落切分 | `splitIntoParagraphs()` | 按空行分割，合并短行 | 史记原文一行一段，需要合并成完整段落 |
| 5. Chunk生成 | `ChunkText()` | 按策略切分，保持重叠 | 确保上下文不丢失 |

**调用的库**：
```go

    "path/filepath"   // 文件路径处理
)
```

### 为什么选这个方案

**场景背景**：我们处理的是《史记》《左传》等古籍和《百家讲坛》讲稿

| 问题 | 我们的做法 | 其他方案对比 |
|------|-----------|-------------|
| 古籍没有段落标记 | 按句号+换行智能合并 | 固定长度切分会切断句子 |
| 需要保留章节结构 | 正则识别"卷X"、"本纪" | 纯向量相似度会丢失结构信息 |
| 不同文档类型粒度不同 | 史书800字，讲座1200字 | 统一512字符不适合长叙述 |
| 需要支持增量更新 | 按Source字段覆盖 | 全量重建成本太高 |

## 2. 多 Agent 之间如何协作

### 当前方案

**澄清：当前项目实际上是单 Agent 架构，不是多 Agent**

**现有架构代码**：`backend/services/agent/agent.go`

### 为什么选单 Agent + Tools 方案

**场景背景**：教学对话助手需要回答历史问题，可能需要搜索知识库、查询图谱、联网搜索

| 方案 | 复杂度 | 适用场景 | 我们的选择 |
|------|--------|---------|-----------|
| 单Agent + Tools | 低 | 任务类型单一，工具调用即可满足 | ✅ 当前方案 |
| 多Agent协作 | 高 | 复杂任务分解（研究→写作→审核） | 未来扩展 |
| 分层Agent | 中 | 需要规划+执行分离 | 暂不需要 |

**不选多Agent的原因**：
1. 教学对话是单线程问答，不需要分解子任务
2. 工具调用（搜索/图谱/联网）足够覆盖需求
3. 多Agent增加延迟和调试复杂度

### 优化计划：引入多 Agent 架构

**场景扩展**：未来要支持"备课助手"，需要多个专业Agent协作

**新架构设计**：

```

    ├── ResearchAgent (研究Agent) - 搜索资料、整理大纲
    ├── WritingAgent (写作Agent) - 生成教案内容
    ├── ReviewAgent (审核Agent) - 检查准确性、补充引用
    └── PPTAgent (PPT生成Agent) - 制作课件
```

### 为什么选这个方案

**场景背景**：教学对话需要长期上下文，但LLM有token限制

| 方案 | 优点 | 缺点 | 我们的选择 |
|------|------|------|-----------|
| 滑动窗口（只保留最近N条） | 简单 | 丢失早期重要信息 | ❌ |
| 全文摘要（每次摘要全部） | 信息完整 | 早期细节丢失 | ❌ |
| 分层摘要（我们的方案） | 保留高重要性消息+摘要 | 复杂度中等 | ✅ |
| 向量记忆（所有消息向量化） | 可检索任意历史 | 检索可能不准确 | 未来补充 |

**关键设计决策**：
1. **保留最近5条**：确保对话连贯性
2. **重要性>=7的消息保留**：关键问答不丢失
3. **摘要包含话题和实体**：支持快速检索

```go
type ContextNode struct {
    Type NodeType  // Message/Summary/Topic
    Level int      // 0=原始消息, 1=一级摘要, 2=二级摘要
    Importance float64  // 动态重要性（基于访问频率更新）
    AccessCount int     // 访问次数
    Children []*ContextNode  // 子节点
}

// 动态重要性计算
func (n *ContextNode) updateImportance() {
    // 时间衰减 + 访问频率
    timeDecay := math.Exp(-float64(time.Since(n.AccessedAt).Hours()) / 24)
    frequency := math.Log(float64(n.AccessCount) + 1)
    n.Importance = baseScore * (0.3*timeDecay + 0.7*frequency)
}
```

```go
// ImportDocument 完整导入流程
func (s *ImportService) ImportDocument(filePath string, metadata DocumentMetadata, docType DocumentType) error {
    // Step 1: 文档解析
    parser := NewDocumentParser(docType)
    chunks, err := parser.ParseFile(filePath, metadata)
    // 输出: []TextChunk, 每个Chunk包含Content和Metadata
    
    // Step 2: 导入MySQL（主存储）
    for _, chunk := range chunks {
        knowledge := &models.KnowledgeChunk{
            Title:     chunk.Metadata.Title,
            Content:   chunk.Content,
            Category:  chunk.Metadata.Category,
            Source:    chunk.Metadata.Source,
            // ...
        }
        dao.Db.Create(knowledge)  // 生成自增ID
    }
    
    // Step 3: 向量化（批量）
    texts := extractTexts(chunks)
    embeddings, _ := s.embedding.BatchGetEmbeddings(texts, 20)
    // 调用OpenAI Embedding API，batch=20，带缓存
    
    // Step 4: 导入Milvus（向量库）
    entities := buildMilvusEntities(chunks, embeddings)
    s.milvus.InsertBatch(entities)
}
```

### 为什么选这个方案

**场景背景**：需要支持语义检索（同义词、相关概念），但MySQL全文检索无法满足

| 方案 | 延迟 | 准确度 | 成本 | 我们的选择 |
|------|------|--------|------|-----------|
| MySQL FULLTEXT | 50ms | 低（关键词匹配） | 低 | 备用方案 |
| Milvus向量检索 | 30ms | 高（语义相似） | 中（需GPU） | ✅ 主方案 |
| Elasticsearch | 40ms | 中（BM25+向量） | 高 | 暂不需要 |
| Pinecone等SaaS | 50ms | 高 | 高（按量付费） | 数据安全考虑 |

**关键技术决策**：

1. **双写架构**：MySQL做主存储（持久化+元数据查询），Milvus做向量检索
2. **OpenAI Embedding**：1536维，平衡质量和成本（$0.02/1M tokens）
3. **AUTOINDEX**：让Milvus自动选择HNSW或IVF索引
4. **Redis缓存**：Embedding结果缓存24小时，检索结果缓存10分钟

**阶段1：多向量模型支持**

```go
// Agent.Run() 的核心是循环决策
func (a *DialogAgent) Run(ctx context.Context, messages []Message, tools []Tool, callback EventCallback) (*AgentResult, error) {
    for round := 0; round < a.config.MaxToolRounds; round++ {
        // 1. 调用LLM生成回复
        response, _ := a.llmClient.StreamChat(messages, tools, ...)
        
        // 2. LLM自主决定是否调用工具
        if len(response.ToolCalls) == 0 {
            // 没有工具调用，直接返回结果
            return &AgentResult{Content: response.Content}, nil
        }
        
        // 3. 有工具调用，执行工具
        for _, tc := range response.ToolCalls {
            result, _ := a.toolExecutor.ExecuteTool(tc.Function.Name, tc.Function.Arguments)
            
            // 4. 工具结果加入上下文，继续循环
            messages = append(messages, Message{
                Role: "tool",
                Content: result,
            })
        }
        // 循环继续，LLM再次决策
    }
}
```

**特点**：
- 每轮由LLM决定是回答还是调用工具
- 工具调用次数不固定（0-5次）

```go
    // 暂停/恢复（长时Skill）
    Pause(skillInstanceID string) error
    Resume(skillInstanceID string, userInput string) error
}

type SkillCallback interface {
    OnProgress(progress float64, message string)
    OnIntermediateResult(result interface{})
    OnNeedUserInput(prompt string) string  // 阻塞等待用户输入
}
```

```go
}

    r.skills[skill.Name()] = skill
}

    // 例如："备课" → [LessonPrepSkill, ResearchSkill]
```

**阶段3：Agent 动态调用 Skill**
```go
func (a *DialogAgent) useSkill(skillName string, input map[string]interface{}) {
    skill := a.skillRegistry.Get(skillName)
    
    result, err := skill.Execute(ctx, input, SkillCallback{
        OnProgress: func(p float64, msg string) {
            // 发送SSE给用户
            a.sendEvent(EventTypeProgress, msg)
        },
        OnNeedUserInput: func(prompt string) string {
            // 暂停Skill，等待用户回复
            a.pauseSkill(skillName)
            a.sendEvent(EventTypeNeedInput, prompt)
            return a.waitForUserInput()
        },
    })
}
```

```go
package prompts

type PromptManager struct {
    templates map[string]*Template
    loader    *TemplateLoader
}

// 加载提示词
type TemplateLoader struct {
    basePath string
}

func (l *TemplateLoader) Load(name string) (*Template, error) {
    // 1. 读取YAML文件
    data, _ := os.ReadFile(filepath.Join(l.basePath, name+".yaml"))
    
    // 2. 解析YAML
    var tmpl Template
    yaml.Unmarshal(data, &tmpl)
    
    // 3. 处理import语句
    tmpl.System = l.processImports(tmpl.System)
    tmpl.User = l.processImports(tmpl.User)
    
    return &tmpl, nil
}

// 渲染提示词
func (t *Template) Render(variables map[string]interface{}) (string, string, error) {
    // system prompt
    systemTmpl := template.Must(template.New("system").Parse(t.System))
    var systemBuf bytes.Buffer
    systemTmpl.Execute(&systemBuf, variables)
    
    // user prompt
    userTmpl := template.Must(template.New("user").Parse(t.User))
    var userBuf bytes.Buffer
    userTmpl.Execute(&userBuf, variables)
    
    return systemBuf.String(), userBuf.String(), nil
}
```

| 层级 | 内容 | 示例 | 复用性 |

| **任务层** | 定义具体任务和输出格式 | "设计课程大纲，包含..." | 中（同类任务共用） |
| **约束层** | 输出格式、安全限制 | "必须按JSON格式输出" | 高（全局共用） |

| **示例层** | Few-shot示例 | "示例：输入X→输出Y" | 中（同类任务共用） |

## 10. Tool 是什么，在项目中是如何体现的

### 概念定义

**Tool（工具）** = 可被 LLM 调用的外部功能接口

组成要素：
1. **描述**（Description）：告诉LLM这个工具能做什么
2. **参数Schema**：定义输入参数的结构
3. **执行逻辑**：实际的功能实现

### 项目中的体现

**Tool 接口定义**：`backend/services/tools/tool.go`

```go
// Tool 标准接口
type Tool interface {
    Info() *ToolInfo                    // 工具元数据
    Run(ctx context.Context, input string) (string, error)  // 执行逻辑
}

// ToolInfo 工具信息
type ToolInfo struct {
    Name        string                 `json:"name"`
    Description string                 `json:"description"`
    Parameters  map[string]interface{} `json:"parameters"`  // JSON Schema
}
```

### 为什么这样设计

**场景背景**：教学对话需要准确的历史知识，但LLM可能幻觉

| 方案 | 优点 | 缺点 | 我们的选择 |
|------|------|------|-----------|
| 纯LLM回答 | 简单 | 可能幻觉，不准确 | ❌ |
| RAG检索后回答 | 有知识库支撑 | 需要手动写检索逻辑 | 部分使用 |
| **Tool调用（我们的方案）** | LLM自主决定何时检索 | 需要定义Tool Schema | ✅ |
| Function Calling | 标准化 | 依赖特定模型支持 | 底层使用 |

**关键设计决策**：
1. **标准化Tool接口**：借鉴OpenAI Function Calling，但解耦具体实现
2. **自描述**：每个Tool通过Info()自描述，LLM自动理解用途
3. **可扩展**：新增Tool只需实现接口，注册即可使用
