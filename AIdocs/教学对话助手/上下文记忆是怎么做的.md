# 教学对话助手 — 上下文记忆实现

## 一、实现方案

本项目采用**数据库持久化 + 窗口截取 + LLM 摘要压缩**的两层记忆方案，不依赖 Redis。

---

## 二、数据层（`services/memory/session.go`）

### `sessions` 表

```sql
CREATE TABLE sessions (
    id         VARCHAR(64) PRIMARY KEY,  -- UUID
    user_id    UINT,
    mode       VARCHAR(20),              -- local / online / auto
    title      VARCHAR(255),             -- 会话标题（首条消息截取）
    created_at DATETIME,
    updated_at DATETIME
);
```

### `messages` 表

```sql
CREATE TABLE messages (
    id          BIGINT AUTO_INCREMENT PRIMARY KEY,
    session_id  VARCHAR(64),  -- INDEX
    role        VARCHAR(20),  -- user / assistant / system / tool
    content     LONGTEXT,
    tool_calls  TEXT,         -- 工具调用 JSON
    tool_result TEXT,         -- 工具返回结果
    created_at  DATETIME
);
```

## 三、窗口记忆（`BuildContext`）

每次请求时，从 DB 取最近 **20 条**消息作为上下文窗口：

```go
func (m *SessionManager) BuildContext(sessionID string, maxMessages int) ([]llm.Message, error) {
    messages, _ := m.GetMessages(sessionID, maxMessages)
    context := make([]llm.Message, 0, len(messages))
    for _, msg := range messages {
        context = append(context, llm.Message{
            Role:    msg.Role,
            Content: msg.Content,
        })
    }
    return context, nil
}
```

调用时取 `maxMessages=20`（`controllers/dialog.go:85`），控制单次请求的 Token 消耗。

---

## 四、会话摘要压缩（`SummarizeHistory`）

当会话消息超过 10 条时触发，将早期消息压缩为摘要：

```go
func (m *SessionManager) SummarizeHistory(sessionID string) (string, error) {
    messages, _ := m.GetMessages(sessionID, 0)
    if len(messages) <= 10 {
        return "", nil  // 不需要压缩
    }

    // 将前 N-5 条消息拼成文本
    var historyText string
    for _, msg := range messages[:len(messages)-5] {
        historyText += fmt.Sprintf("%s: %s\n", msg.Role, msg.Content)
    }

    // 调用 LLM 压缩
    prompt := "请将以下历史对话内容压缩成简洁的摘要，保留关键信息：\n\n" + historyText
    resp, _ := m.llmClient.Chat([]llm.Message{{Role: "user", Content: prompt}}, nil)
    summary := resp.Choices[0].Message.Content

    // 摘要以 system role 写回 DB，供后续轮次的 BuildContext 使用
    m.AddMessage(sessionID, "system", "[对话摘要] "+summary, "", "")
    return summary, nil
}
```

摘要存储为 `role=system` 的消息，`BuildContext` 读取时会自然包含在上下文中，且 LLM 会将 system 消息视为背景信息。

## 五、会话标题自动生成

```go
func (m *SessionManager) GenerateTitle(sessionID string, firstMessage string) error {
    title := firstMessage
    if len(title) > 20 {
        title = title[:20] + "..."
    }
    return m.UpdateTitle(sessionID, title)
}
```

首次对话时（`req.SessionID == ""`）触发，截取前 20 字符，写入 `sessions.title`。

## 六、上下文注入 Prompt

`controllers/dialog.go` 中组装最终 messages 的顺序：

```
[system]   buildSystemPrompt（角色定义 + mode 说明）
[user]     历史消息1（来自 BuildContext）
[assistant] 历史消息2
...
[system]   [对话摘要] xxxxxx（若已触发摘要压缩）
[user]     本轮用户问题（已通过 AddMessage 写入 DB，包含在 BuildContext 中）
```

## 七、与 Redis 摘要方案的对比

| 维度 | 本项目（DB 全量持久化）| 文档中描述的 Redis 方案 |
|------|----------------------|----------------------|
| 持久化 | MySQL 全量 | MySQL 归档 + Redis 热缓存 |
| 查询速度 | DB 查询（有 session_id 索引） | Redis 读（更快）|
| 跨重启 | 数据不丢失 | Redis 需持久化配置 |
| 摘要触发 | `len(messages) > 10` | Token 超阈值 + 每 N 轮 |
| 适用场景 | 当前规模 | 高并发大量用户场景 |

> 当前实现以 DB 为主，满足功能需求；若并发量提升，可在 `GetMessages` 前加 Redis 缓存层。
